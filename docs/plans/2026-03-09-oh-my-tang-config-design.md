# oh-my-tang Config File Design

**Date:** 2026-03-09

**Summary**

Add first-class `.oh-my-tang.json` support so the plugin can persist user-editable runtime settings near `opencode.json` instead of relying on hard-coded defaults in `src/plugin.ts`.

**Goals**

- Auto-generate `.oh-my-tang.json` during plugin initialization when it is missing.
- Prefer creating the file in the same directory as `opencode.json`.
- Fall back to the worktree root when `opencode.json` cannot be found.
- Resolve runtime config with this priority: built-in defaults < `.oh-my-tang.json` < environment overrides.
- Surface config path, source, generation status, and warnings through `tang_config`.

**Non-Goals**

- Adding a CLI for editing Tang config.
- Supporting multiple config file names.
- Moving persisted execution state out of `.tang-dynasty/state.json`.
- Introducing environment variables for every config field in the first version.

**User Experience**

On plugin startup:

1. Search for `opencode.json` starting from `input.worktree`.
2. If found, use its directory as the config directory.
3. If not found, use `input.worktree`.
4. Ensure `.oh-my-tang.json` exists there.
5. Load config from the file, merge it with defaults, then apply environment overrides.

The generated file should contain the editable operational settings:

```json
{
  "maxConcurrentMinistries": 3,
  "maxReviewRounds": 3,
  "tokenBudgetLimit": 100000,
  "healthRiskProfile": "balanced",
  "enableParallelExecution": true,
  "verbose": false
}
```

**Schema**

First version supports:

- `maxConcurrentMinistries: number`
- `maxReviewRounds: number`
- `tokenBudgetLimit: number`
- `healthRiskProfile: "balanced" | "strict" | "relaxed"`
- `enableParallelExecution: boolean`
- `verbose: boolean`

Invalid values should be ignored field-by-field, not all-or-nothing.

**Behavior Rules**

- Missing file: create it with defaults.
- Missing `opencode.json`: create `.oh-my-tang.json` in the worktree root and emit a warning in `tang_config`.
- Invalid JSON: keep running with defaults, preserve the broken file unchanged, emit a warning in `tang_config`.
- Invalid field values: ignore those fields, keep valid ones, emit warnings in `tang_config`.
- Write failure: keep running with defaults and emit a warning in `tang_config`.
- `TANG_HEALTH_RISK_PROFILE` continues to override config-file `healthRiskProfile`.

**Implementation Shape**

Add a dedicated config helper module instead of putting file discovery, JSON parsing, validation, generation, and warning aggregation into `src/plugin.ts`.

Recommended pieces:

- `resolveTangConfig(...)` helper to find/generate/load/validate config.
- Config-file metadata structure for:
  - `path`
  - `directory`
  - `source`
  - `autoGenerated`
  - `warnings`
  - `foundOpencodeConfig`
- `TangConfigReport` extension to expose config-file state to operators.

**Testing**

Add regression coverage for:

- auto-generation beside `opencode.json`
- fallback generation at the worktree root
- loading valid file overrides
- preserving env override precedence
- invalid JSON warning behavior
- invalid field warning behavior

**Risks**

- The current repository snapshot is missing some imported runtime/agent files, so broader test execution may still be blocked outside the scoped tests for this feature.
- Config-file discovery should stay deterministic and not walk outside the active worktree.

**Decision**

Implement `.oh-my-tang.json` auto-generation at plugin initialization, with config file placement tied to the directory containing `opencode.json` when available.
